1 · Purpose and Scope

This ceremony generates the Root CA private key inside a (no HSM) device and establishes a wrap key for key backup and recovery.

This ceremony is performed in accordance with ETSI EN 319 401 V3.2.1 §7.5 (Cryptographic Controls) and §7.6 (Physical and Environmental Security). The Shamir Secret Sharing implementation is ssss (B. Poettering). Share encryption uses age with age-plugin-yubikey (str4d).

⚠ This is a security-sensitive ceremony. All commands shall be executed only by the designated Operator in the presence of all custodians and witnesses. No part of this ceremony shall be performed outside the designated secure room. All mobile phones and networked devices must be removed before the ceremony begins.

Verify Hardware RNG Device

The signing key will be generated externally using a dedicated hardware random number generator. The RNG device must be identified and tested before use.

# ── External RNG Device Verification ──
# RNG device: /dev/ttyUSB0

# Verify the device exists and is readable
test -r /dev/ttyUSB0 && echo "RNG device /dev/ttyUSB0: accessible ✓" || echo "FAIL: /dev/ttyUSB0 not found or not readable"

# Identify the hardware RNG
cat /sys/class/misc/hw_random/rng_current 2>/dev/null || echo "(kernel hw_random not applicable)"

  • RNG device /dev/ttyUSB0 present and readable
  • Hardware RNG identity recorded in ceremony log
Step 5
RNG Entropy Quality Tests

Before using the hardware RNG for key generation, statistical tests confirm the entropy source produces sufficient randomness. All tests must pass.

# ── RNG Entropy Quality Tests ──
# Collect a test sample from the RNG device
dd if=/dev/ttyUSB0 of=/tmp/ceremony/rng-sample.bin bs=1024 count=256 2>/dev/null
echo "Collected 256 KiB sample for entropy testing"

# Test 1: ent — basic entropy estimation
# Ideal: entropy ≈ 7.999 bits/byte, chi-square 1-99%
ent /tmp/ceremony/rng-sample.bin

# Test 2: rngtest (from rng-tools) — FIPS 140-2 statistical tests
# Runs Monobit, Poker, Runs, and Long Runs tests
# Acceptable: 0 failures in 1000+ blocks
dd if=/dev/ttyUSB0 bs=2500 count=101 2>/dev/null | rngtest --blockcount=100

# Test 3: Compression ratio — truly random data is incompressible
ORIG=$(wc -c < /tmp/ceremony/rng-sample.bin)
gzip -c /tmp/ceremony/rng-sample.bin | wc -c | awk -v orig="$ORIG" '{printf "Compression ratio: %.4f (ideal ≈ 1.00)\\n", $1/orig}'

# Test 4: Check for stuck bits (compare two independent samples)
dd if=/dev/ttyUSB0 of=/tmp/ceremony/rng-sample2.bin bs=1024 count=256 2>/dev/null
if cmp -s /tmp/ceremony/rng-sample.bin /tmp/ceremony/rng-sample2.bin; then
  echo "FAIL: Two samples are identical — RNG may be stuck!"
  exit 1
else
  echo "CONFIRMED: Samples differ ✓"
fi

# Clean up test samples
shred -u /tmp/ceremony/rng-sample.bin /tmp/ceremony/rng-sample2.bin
  • ent: entropy ≥ 7.99 bits/byte
  • ent: chi-square between 1% and 99%
  • rngtest: FIPS 140-2 tests — 0 failures
  • Compression ratio ≥ 0.99 (incompressible)
  • Two independent samples confirmed different
⚠ If any test fails, STOP the ceremony. Do not proceed with key generation. Replace or investigate the RNG device before restarting.

6 · Secret Generation

The Root CA private key is generated outside the HSM using a dedicated hardware RNG (/dev/ttyUSB0), which was verified in Section 4. The key will be imported into the (no HSM) after generation. A wrap key is also generated from the same RNG for encrypted key backup.

Step 6a
Generate Root CA Key from Hardware RNG 
# Generate EC P-384 private key using hardware RNG: /dev/ttyUSB0
# Key is generated OUTSIDE the HSM and will be imported after wrapping.
openssl ecparam -genkey -name secp384r1 -rand /dev/ttyUSB0 -noout -out ca-key.pem

# Extract public key
openssl ec -in ca-key.pem -pubout -out root-ca-pubkey.pem
cat root-ca-pubkey.pem

# Verify key is valid
openssl ec -in ca-key.pem -check -noout
echo "Example Root CA G1 signing key generated from external RNG ✓"

# Convert private key to DER for HSM import
openssl ec -in ca-key.pem -outform DER -out ca-key.der

# The private key will be imported into the HSM and then securely erased from RAM.
  • EC P-384 key generated from /dev/ttyUSB0
  • Public key exported and recorded
  • Key validity confirmed by OpenSSL
Step 6b
Generate Wrap Key from Hardware RNG 
# Generate 256-bit (32-byte) wrap key from hardware RNG
# Source: /dev/ttyUSB0 (tested above)
WRAP_KEY=$(dd if=/dev/ttyUSB0 bs=32 count=1 2>/dev/null | xxd -p | tr -d '\n')

# Verify length — must be exactly 64 hex characters
echo "Key length (chars): $(echo -n "${WRAP_KEY}" | wc -c)"
# Expected: 64

# Show only first 4 and last 4 hex chars for witness confirmation
# The full key is NEVER displayed on screen (camera is recording)
echo "WRAP KEY fingerprint: ${WRAP_KEY:0:4}....${WRAP_KEY: -4}"

# Source confirmed: hardware RNG /dev/ttyUSB0
  • Key length confirmed: 64 hex characters (256 bits)
  • First 4 and last 4 hex chars confirmed by all witnesses
Step 6c
Generate Self-Signed Certificate

The externally generated private key is used to create a self-signed certificate.

# Create openssl.cnf to be used while generating certificate request
cat > openssl.cnf << EOF
[ v3_ca ]
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid:always,issuer
basicConstraints = critical,CA:true
EOF

# Generate certificate request
openssl req -new -key ca-key.pem -out ca-cert.req -subj '/CN=Sweden Connect Trust Anchor/O=Sweden Connect/C=SE'

# Generate self-signed certificate
openssl x509 -req -days 7300 -in ca-cert.req -signkey ca-key.pem -out ca-cert.pem \
   -extfile openssl.cnf -extensions v3_ca
  • openssl.cnf created
  • Certificate request created
  • Self-signed certificate created
Step 6d
Export encrypted CA Key into USB

The externally generated private key is exported into a USB stick. After export, the plaintext key is securely erased from the workstation.

# Export externally generated key into USB
echo ${WRAP_KEY} | openssl ec -aes256 -in ca-key.pem \
  -out ca-key-backup.pem -passout stdin

# Securely erase the plaintext private key
shred -u ca-key.pem ca-key.der
echo "Private key securely erased from workstation ✓"
  • Plaintext key securely erased from workstation RAM

Copy ca-key-backup.pem to a dedicated USB drive. Eject the drive and place in a separate tamper-evident bag.

7 · Shamir Secret Sharing — Split (2-of-3)

⚠ All 3 custodians must be present during this step. Share values are written directly to individual files and are never displayed on screen. This is the most sensitive moment of the ceremony.
Step 7
Split Secret into 3 Shares 
# Split wrap key into 3 shares, threshold 2
# -t 2  = minimum shares required to reconstruct
# -n 3  = total shares generated
# -x   = hex input/output mode (required for binary secrets)
# -s 256 = prime field size in bits (must be >= secret bit length)
# Output is written directly to individual share files (never shown on screen)
echo -n "${WRAP_KEY}" | ssss-split -t 2 -n 3 -x -s 256 | while IFS= read -r line; do
  NUM=$(echo "$line" | cut -d- -f1)
  echo "$line" > "share-${NUM}.txt"
  echo "Share ${NUM} written to share-${NUM}.txt"
done

# Confirm all 3 share files exist
ls -1 share-*.txt | wc -l
# Expected: 3

9 · Verification

⚠ All custodians must remain in the room during verification.

Before destroying plaintext material, verify that 2 shares reconstruct the original secret.

Step 9a
Test Reconstruction with 2 Shares 
# Decrypt test shares — each participating custodian inserts their YubiKey
# age-plugin-yubikey will prompt for PIV PIN

# Custodian 1 — insert YubiKey, then:
age-plugin-yubikey --identity | age --decrypt -i - custodian1.share.age > share1.txt

# Custodian 2 — insert YubiKey, then:
age-plugin-yubikey --identity | age --decrypt -i - custodian2.share.age > share2.txt

# Combine the 2 decrypted shares
cat share1.txt share2.txt | ssss-combine -t 2 -x > reconstructed.txt

# Automated comparison — secrets are NOT displayed on screen
if [ "$(cat reconstructed.txt)" = "${WRAP_KEY}" ]; then
  echo "VERIFICATION PASSED — reconstructed key matches original"
else
  echo "VERIFICATION FAILED — keys do not match"
fi

# Wipe reconstruction artifact
shred -u reconstructed.txt
  • Reconstructed secret matches original wrap key — confirmed by automated comparison (no secrets shown on screen)
  • Test share files wiped immediately: shred -u share*.txt
